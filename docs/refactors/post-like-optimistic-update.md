## 목적

이 리팩터의 핵심은 "클릭 후 즉시 반응"과 "서버 정합성"을 동시에 지키는 것이다. 좋아요는 사용자가 누른 순간 피드백이 와야 하는 상호작용이므로, 긴 `setTimeout` 기반의 가짜 대기 시간보다 짧은 optimistic update가 낫다. 가짜 지연은 네트워크가 빠른 환경에서도 일부러 느리게 보이게 만들고, 실패/롤백과 무관한 체감 지연만 키운다.

## 현재 코드 현실

### `useTogglePostLikeMutation.ts`

- `useMutation`의 `mutationFn`은 `togglePostLike(postId)`를 호출한다.
- `onMutate`에서 `queryKeys.post.detail(postId)`를 취소하고, 캐시가 있으면 `likedByMe`를 토글한 뒤 `likeCount`를 `+1/-1`로 즉시 바꾼다.
- 이전 상세 데이터는 `previousPost`로 저장한다.
- `onError`는 `previousPost`가 있을 때만 상세 캐시를 복구한다.
- `onSettled`는 `queryKeys.post.all`과 해당 상세 쿼리를 모두 무효화한다.

### `PostPage.tsx`

- `postId`를 숫자로 파싱하고, 양의 정수일 때만 상세 쿼리를 활성화한다.
- `usePostDetailQuery` 결과를 `mapPostDetailToPost`로 변환해 `PostDetailCard`에 넘긴다.
- 좋아요 클릭 시 `mutateAsync`를 호출하고, 실패하면 401은 로그인 토스트, 그 외는 일반 실패 토스트를 보여준다.
- `isPending` 상태를 카드에 전달해 버튼을 비활성화한다.

### `PostDetailCard.tsx`

- `likedByMe`에 따라 하트 버튼 variant가 바뀐다.
- `heart`는 좋아요 카운트 표시값으로 사용된다.
- `disabled || isLikePending`이면 하트 버튼이 막힌다.
- 카드 전체는 `disabled`일 때 시각적으로 비활성화된다.

## 현재 리스크

### 1. 짧은 optimistic window보다 긴 지연을 쓰면 UX가 나빠진다

좋아요는 즉시성에 민감한 액션이다. `setTimeout`으로 "잠깐 반응을 늦춘 뒤" 상태를 바꾸는 방식은 실제로는 서버를 더 잘 다루는 것도 아니고, 사용자가 눌렀다는 사실을 감춘다. optimistic update는 클릭 직후 UI를 바꾸고, 실패했을 때만 되돌리므로 상호작용 비용이 훨씬 낮다.

### 2. 캐시가 없으면 즉시 반영이 사라진다

현재 optimistic update는 상세 캐시가 존재할 때만 동작한다. 캐시가 없으면 서버 요청은 가지만 즉시 UI 반영은 없다. 이 경우에도 실패/성공 후 정합성은 맞지만, 사용자가 기대하는 "즉시 반응"이 약해진다.

### 3. `invalidateQueries({ queryKey: post.all })`가 넓다

좋아요 하나를 눌렀는데 목록 전체를 다시 읽어오면, 데이터가 많을수록 불필요한 네트워크와 깜빡임이 생긴다. 현재는 안전 우선 전략이지만, 실제로 좋아요 수가 표시되는 범위를 기준으로 더 좁혀야 한다.

### 4. 상세 롤백만 있고 다른 캐시와의 동기화 경계가 약하다

상세 페이지 캐시는 복구되지만, 같은 게시글이 목록 카드나 추천 영역에도 노출된다면 그 쪽 캐시는 별도로 갱신되어야 한다. 지금처럼 "일단 전부 무효화"는 맞지만 비용이 크다.

## 권장 상태 흐름

1. 버튼 클릭.
2. `PostPage`에서 유효한 `postId`일 때만 mutation 실행.
3. `onMutate`에서 상세 캐시를 즉시 토글한다.
4. UI는 pending 상태를 보여주되, 가짜 지연은 넣지 않는다.
5. 서버 성공 시에는 현재 캐시를 기준으로 다시 정합성을 확인한다.
6. 실패 시에는 `previousPost`로 정확히 롤백한다.
7. settle 이후에는 필요한 범위만 무효화한다.

이 흐름의 핵심은 "낙관적 반영은 즉시, 네트워크 정합성은 나중"이다. 사용자 체감은 빠르게 유지하고, 서버 결과는 최종 정답으로 수렴시킨다.

## 오류 처리 규칙

- 401은 인증 문제로 분류하고, 좋아요 UI는 롤백한 뒤 로그인 안내만 보여준다.
- 네트워크 오류, 5xx, 알 수 없는 오류는 공통 실패 토스트로 처리한다.
- 실패 메시지는 서버에서 오는 경우 우선 사용하되, 없으면 공통 문구를 사용한다.
- rollback은 토스트보다 먼저 적용되어야 한다. 사용자가 에러 토스트를 보는 순간 화면도 이미 원래 상태로 돌아와 있어야 한다.

## 캐시 무효화 경계

### 반드시 유지할 것

- 상세 캐시 `post.detail(postId)`는 최종 정답을 위해 정리한다.
- 게시글 목록 계열 `post.all`은 실제로 좋아요 수/하트 상태를 보여주는 화면이 있을 때만 무효화한다.

### 가능하면 좁힐 것

- 목록 전체를 무효화하기보다, 좋아요가 노출되는 특정 리스트 쿼리로 범위를 한정한다.
- 같은 게시글이 여러 화면에 나타난다면, "현재 화면 + 실제 노출 화면"만 갱신한다.

### 판단 기준

- 화면이 직접 좋아요 상태를 보여주면 캐시 갱신 대상이다.
- 화면이 좋아요 상태를 보여주지 않으면 무효화할 이유가 없다.

## 엣지 케이스

- 캐시 없는 상세 진입 직후 좋아요 클릭
- 빠른 연속 클릭으로 인한 요청 중복
- 401 발생 후 토스트와 롤백 순서
- 서버 성공 후 목록과 상세의 값 불일치
- `likeCount`가 0인 상태에서 취소 요청
- 네트워크 지연 중 다른 쿼리 갱신과 충돌
- 컴포넌트 언마운트 후 응답 도착

## 테스트 체크리스트

- 좋아요 클릭 즉시 하트와 카운트가 바뀌는지 확인한다.
- 실패 시 이전 값으로 롤백되는지 확인한다.
- 401에서 로그인 토스트가 뜨고 상태가 복구되는지 확인한다.
- pending 동안 추가 클릭이 막히는지 확인한다.
- `likeCount`가 음수로 내려가지 않는지 확인한다.
- 상세 화면과 목록 화면이 같은 게시글 상태를 공유하는지 확인한다.
- 네트워크 성공 후 캐시 무효화가 필요한 범위만 다시 읽는지 확인한다.
- 화면을 빠르게 이동해도 경고나 잘못된 업데이트가 남지 않는지 확인한다.

## 롤아웃 순서

1. 현재 mutation의 optimistic update와 rollback 동작을 먼저 유지한 채 긴 지연이 있다면 제거한다.
2. pending 상태를 버튼 단위로만 막고, 전체 카드 비활성화는 필요한 경우에만 사용한다.
3. 무효화 범위를 실제 노출 화면 기준으로 좁힌다.
4. 상세/목록이 같은 like 상태를 읽는지 확인하고, 누락된 화면이 있으면 거기까지 포함해 정리한다.
5. 실패 토스트와 롤백 순서를 브라우저에서 검증한다.

## 비목표

- 서버 API 스펙 변경
- 좋아요 정책 변경
- 배지/카운트 디자인 변경
- 댓글 입력과의 상호작용 변경
- 전역 상태 관리 도입
- 오프라인 큐잉 또는 재시도 전략 추가

## 결론

이 리팩터는 "더 자연스럽게 빠른 반응"을 만드는 작업이다. 핵심은 delay를 흉내 내는 것이 아니라, 클릭 즉시 optimistic update를 하고 실패 시에만 되돌리는 구조를 지키는 것이다. 현재 구조는 이미 그 방향에 가깝기 때문에, 다음 단계는 더 넓은 무효화 대신 더 좁은 정합성 경계와 더 명확한 에러 복구 순서를 만드는 것이다.
