# Release Guide

## 목적
이 문서는 `dev -> main` 릴리즈를 안정적으로 진행하기 위한 공통 규칙을 정의한다.

- 릴리즈 PR 형식 통일
- 배포 전 검증 기준 통일
- 롤백 및 모니터링 기준 사전 합의

---

## 릴리즈 브랜치/PR 운영 규칙

### 1) 브랜치 전략
1. `dev` 기준으로 릴리즈 브랜치 생성
- 예시: `release/v1.2.0`
2. 릴리즈 브랜치에서는 신규 기능 추가를 금지하고 버그 수정만 허용
3. 릴리즈 브랜치에서 `main`으로 Release PR 생성
4. 머지 후 태그 생성
- 예시: `v1.2.0`
5. `main` 변경사항을 `dev`로 동기화

### 2) 버전 규칙
- `major`: 호환성 깨지는 변경(브레이킹 체인지)
- `minor`: 하위 호환 기능 추가
- `patch`: 버그 수정, 내부 개선

### 3) Release PR 작성 규칙
- `.github/PULL_REQUEST_TEMPLATE/release.md` 사용
- 포함 범위(PR/기능/버그)를 누락 없이 기록
- 배포 전 체크 항목은 실제 실행 후 체크

---

## 배포 전 필수 체크 기준

아래는 Release PR 템플릿의 체크리스트를 보완한 상세 기준이다.

### 1) 설치/정적 검증
- `pnpm install --frozen-lockfile`
- `pnpm run lint`
- `pnpm run test`
- `pnpm run build`

### 2) 수동 QA 최소 시나리오
- 인증: 로그인/로그아웃/회원가입 플로우
- 게시글: 목록/상세/작성/수정/삭제/신고
- 예외: 잘못된 URL, 권한 없는 접근, 네트워크 오류

### 3) 환경값 검증
- `.env`/배포 환경 변수 존재 여부
- API Base URL, SSO URL, 도메인 설정
- 시크릿 누락/오타 여부

### 4) API 연동 검증
- 성공 케이스와 실패 케이스를 모두 확인
- 401/403/404/500 상태 코드별 사용자 안내 확인

## 배포 계획에는 무엇을 써야 하나?

Release PR의 `배포 계획` 섹션은 “언제/어떻게/누가/문제 시 어떻게 복구하는가”를 명확히 쓰는 영역이다.

최소 아래 항목을 포함한다.

### 1) 배포 순서
- 어느 환경부터 반영하는지
- 예시:
  1. `main` 머지 후 CI 성공 확인
  2. 프로덕션 배포 실행
  3. 배포 완료 후 핵심 시나리오 스모크 테스트

### 2) 예상 영향 시간
- 서비스 영향 가능 시간대(예: 10~15분)
- 사용자 공지 필요 여부

### 3) 모니터링 포인트
- 예시:
  - 5xx 비율 급증 여부
  - 로그인/회원가입 실패율
  - 게시글 작성/조회 API 실패율
  - 주요 페이지 렌더 오류(ErrorBoundary 노출 여부)

### 4) 롤백 기준(권장)
- 예시:
  - 5xx 비율이 기준치(예: 5%) 이상 5분 지속
  - 로그인/작성 핵심 플로우 실패 재현

### 5) 롤백 절차(권장)
- 예시:
  1. 이전 배포 버전으로 롤백
  2. 캐시 무효화/재배포 확인
  3. 정상화 확인 후 원인 분석 이슈 생성

---

## Release PR 작성 예시 (요약)

```md
### 5. 배포 계획
- 배포 순서:
  1. main 머지 + CI green 확인
  2. 프로덕션 배포 실행
  3. 홈/게시글/로그인 플로우 스모크 테스트
- 예상 영향 시간: 10분 내외
- 모니터링 포인트:
  - 인증 API(401/403) 비율
  - 게시글 조회/작성 API 5xx 비율
```

---

## 운영 팁
- Release PR은 기능 PR보다 설명을 더 구체적으로 작성한다.
- 체크리스트는 “수행 예정”이 아니라 “수행 완료” 기준으로 체크한다.
- 릴리즈 중 발견된 이슈는 즉시 별도 hotfix 이슈/브랜치로 분리한다.
